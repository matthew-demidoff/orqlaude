## Summary

The subsystem is small, clean, and well-commented ‚Äî the architecture (zero-dep raw fetch, long-poll, per-project cursor, mode-600 config) is sound for a personal-fleet tool. However there are real correctness and security issues that should be addressed before this is documented as "ready for daily use." The highest-priority items, in order:

1. **Token file is briefly world-readable.** Mode 600 is applied *after* the rename, leaving a window where the file (and the tmp file before it) is whatever the umask says (typically 0644 on macOS). A local user can read the token during the race.
2. **Markdown injection breaks delivery.** User-supplied strings (plan `rootTask`, task `title`, note `text`, `exitReason`, PR URLs) are interpolated into `parse_mode: "Markdown"` messages with no escaping. A single `_` in a branch name or `*` in a commit subject causes the entire `sendMessage` to fail with `400 Bad Request: can't parse entities`. The notifier swallows this silently, so notifications just *stop appearing* for affected plans.
3. **Cross-process `StateStore` lock is not real.** `StateStore.writeLock` is a per-instance in-memory promise. The bot creates fresh `StateStore` instances per command, the notifier creates one per tick, and the orqlaude MCP server (the actual fleet driver) has its own instance in another process. Read-then-write from `/kill` can lose updates to a concurrent MCP write (or vice-versa). The tmp+rename gets you atomic *file* writes, not atomic *transactions*.
4. **First-tick notification flood.** When the bot starts against an existing project, the notifier's cursor is empty, so every previously-completed task and previously-collected plan is announced as "just happened." Notes have a backfill (`lastNoteId` is seeded to the latest); task/plan statuses do not.
5. **`SIGINT` handler short-circuits cleanup.** The handler sets `stopFlag.stopped = true` but immediately calls `process.exit(0)`, so the `await notifierTask` and any in-flight `update()` writes (with their `.tmp` files) can be cut mid-flight, leaving orphaned tmp files in `.orqlaude/`.

Details below by file. Severity tags: üî¥ must-fix, üüÝ should-fix, üü° nit / defensive.

## `src/telegram/config.ts`

### üî¥ Token-file permission race (`saveConfig`, lines 54-60)

```ts
await fs.writeFile(tmp, JSON.stringify(cfg, null, 2));
await fs.rename(tmp, CONFIG_PATH);
await fs.chmod(CONFIG_PATH, 0o600);
```

The `writeFile` and `rename` happen with **default umask permissions** (0644 on a stock macOS install). For the duration of those two operations, the bot token sits in a file readable by any local user. The `chmod` after the rename doesn't retroactively close the window.

The tmp filename `${CONFIG_PATH}.${process.pid}.tmp` is also predictable. On a multi-user box, a local attacker who can guess the pid (trivially via `ps`) could pre-create that path as a symlink to e.g. `/tmp/steal-token` and `writeFile` would happily follow it. Practical exploitability is low because `~/.orqlaude/` is normally 0755 with owner-only meaningful contents, but the principle still applies.

**Fix:** create the file with mode 0o600 from the start:

```ts
await fs.writeFile(tmp, JSON.stringify(cfg, null, 2), { mode: 0o600, flag: "wx" });
await fs.rename(tmp, CONFIG_PATH);
```

`mode: 0o600` is honored by the underlying `open(2)` (subject to umask), and `flag: "wx"` (= `O_CREAT | O_EXCL`) defeats the symlink attack. A defense-in-depth follow-up would be to chmod 0o700 on `CONFIG_DIR` in `mkdir`.

Same pattern applies in `src/telegram/notifier.ts` `saveCursor` (lines 59-65), although the cursor file is non-secret.

### üü° `chatId === userId` assumption

`WhitelistEntry.chatId` is always set to `userId` (in CLI `cmdTgWhitelist`, in `/whitelist` Telegram handler, and in `cmdTgTest` indirectly). For private DMs this happens to be correct (Telegram's private-chat IDs equal the user ID), but the type implies the two are independent. If you ever want to support group notifications (negative chat IDs), the current code can't represent that ‚Äî and the stored `chatId` field gives the false impression that it can. Either drop the field or actually populate it from `msg.chat.id` on the first interaction.

### üü° No partial-config recovery

`loadConfig` does `{ ...EMPTY, ...parsed }` which means if the JSON file is corrupted (truncated/invalid JSON), `JSON.parse` throws and the bot crashes on startup. A try/catch around the parse that falls back to a warning + empty config would make the bot self-healing.

## `src/telegram/api.ts`

### üüÝ No backoff for `429 Too Many Requests` / `5xx`

```ts
const res = await fetch(url);
if (!res.ok) throw new Error(`getUpdates ${res.status}: ${await res.text()}`);
```

The bot loop catches the throw, sleeps a flat 5s, and retries. Telegram returns 429 with a `retry_after` field in the JSON body (`parameters.retry_after`), which the current code ignores. On a real rate-limit you'll keep hammering at 0.2 Hz, which is fine for `getUpdates` (long-polled, low rate) but bad for the notifier path that may issue many `sendMessage` calls in a single tick.

**Fix sketch:** detect status 429, parse `parameters.retry_after`, sleep at least that long. For other 5xx, exponential backoff capped at e.g. 60s.

### üüÝ Token leak surfaces

The token is concatenated into the URL via `endpoint(method)`. The error strings here (`getUpdates ${status}: ${body}`, `sendMessage ${status}: ${body}`) intentionally avoid including the URL, which is good. However:

- Node's `fetch` on a hard network error throws `TypeError: fetch failed` with the request URL hidden in `error.cause`. We only log `.message`, so we don't leak in *normal* logging, but any future `JSON.stringify(err)` or `err.stack` would. Worth a one-line comment in `api.ts` warning future maintainers off `console.error(err)` / `err.stack`.
- `getMe`'s URL is logged inside Node's libcurl / undici on `--trace-warnings`. If anyone debugs with `NODE_DEBUG=http`, the token will be in the trace. This is an inherent property of Telegram's URL-in-path token scheme; nothing to fix in our code, but worth documenting.

### üü° No timeout on `fetch`

`getUpdates` long-polls with a 30 s server-side timeout; the client `fetch` has no `AbortController`. If the connection hangs (e.g. NAT silently drops it), the bot will sit forever and the notifier will keep ticking but nobody will read messages. Add an `AbortSignal.timeout(timeoutSec * 1000 + 10_000)` to bound it.

## `src/telegram/commands.ts`

### üî¥ `/kill` races with concurrent MCP writes

`/kill` does `stateStore.update(...)` against a *new* `StateStore` instance scoped to `${projectDir}/.orqlaude`. The MCP server running in another process holds its *own* `StateStore`. The in-memory `writeLock` in `state.ts:116` serializes only the calls that share an instance ‚Äî it gives **no protection across processes**. The classic lost-update sequence:

1. MCP-server `checkin` reads state into memory.
2. Bot `/kill` reads state into memory.
3. MCP server writes (rename) ‚Äî tokens updated, task status `running`.
4. Bot writes (rename) ‚Äî clobbers the MCP write; `stopRequested` is now set but `tokens` regressed to the pre-MCP-write value.

This is not theoretical: the notifier ticks every 5 s and the MCP server writes on every JSONL update, so the windows overlap. The tradeoff comment in `state.ts:8-12` ("atomic JSON writes via tmp+rename are sufficient for the concurrency we expect (handful of MCP calls per second)") is exactly wrong for cross-process writers ‚Äî atomic file writes don't compose into atomic transactions.

**Options, in increasing order of effort:**

1. **Quick:** advisory file lock via `proper-lockfile` or a hand-rolled `O_EXCL` lockfile in `update()`. Adds ~5 ms per write; trivially correct.
2. **Medium:** read-modify-write with mtime/size check before rename; if changed since `load`, reload and retry the mutator (optimistic concurrency). Requires mutators to be pure.
3. **Right thing:** SQLite. The schema-v2 comment already acknowledges this as the eventual answer.

Even option 1 closes the bug. This should be done before users start relying on `/kill` from Telegram to interrupt expensive fleets ‚Äî silently undoing a stop is the worst possible failure mode for a kill button.

### üî¥ Markdown injection / silent send failures

Every message that uses `parseMode: "Markdown"` interpolates raw user data:

- line 81: `truncate(plan.rootTask, 100)` ‚Äî plan root task text
- line 90: `truncate(p.rootTask, 50)` ‚Äî same
- line 102: task title via `renderStatus`
- line 121: `truncate(p.rootTask, 80)` and task titles inside `renderStatus`
- line 135: `truncate(taskTitle, 30)` and `truncate(n.text, 200)` ‚Äî note text
- line 137: `${planId.slice(0, 8)}` ‚Äî safe (UUID prefix) but the format is loose
- `notifier.ts:81, 102, 104, 121`: task titles, exit reasons, note text

Telegram legacy Markdown is unforgiving ‚Äî an unbalanced `_`, `*`, `` ` ``, or `[` causes `sendMessage` to fail with 400. Two consequences:

- `commands.ts` returns "error: ‚Ä¶" to the chat when the throw bubbles up, so the user gets *something* but it leaks the underlying API error string.
- `notifier.ts:144-146` catches and writes a stderr line; the notification is **silently lost** and the cursor has *already* advanced (line 137 happens before the loop at line 140). So Markdown-broken notes never reach the user, ever.

This is going to bite in practice because:
- Branch names commonly contain `_` (e.g. `feature_x_y`).
- Commit subjects often contain `*` or `` ` `` (formatted as code).
- PR titles can contain `[scope]`.
- Note text written by sub-agents will frequently contain code with backticks.

**Fix:** escape user-supplied strings before injection. Two reasonable approaches:

1. **Use `MarkdownV2` everywhere and escape `_*[]()~\`>#+-=|{}.!`** (per Telegram spec).
2. **Use `HTML` parse mode** and escape `&<>` ‚Äî much simpler.

For a defensive baseline today, even a one-liner that strips Markdown control chars from user text before interpolation would prevent the silent-delivery-failure mode.

The `/show` command at line 115 wraps the plan JSON in ` ```json ‚Ä¶ ``` ` ‚Äî if any user-supplied text in the JSON contains a literal triple-backtick, the same failure occurs. Trapping the catch and falling back to `parse_mode: undefined` would at least guarantee delivery.

### üüÝ `/start` is reachable by anyone ‚Äî DoS / spam

```ts
if (text === "/start") {
  await api.sendMessage(chatId, `‚Ä¶ Your Telegram user id: \`${userId}\` ‚Ä¶`);
  return;
}
```

This is correct in spirit (the user needs *some* way to discover their own ID), but:

- It runs **before** any rate limiting or whitelist check.
- It does an unconditional `sendMessage`, which costs an HTTP round-trip per invocation.
- A determined attacker who finds your bot's username can spam `/start` and (a) consume your bot's Telegram rate budget, (b) cause the long-poll loop to spend its budget on replies instead of legitimate updates.

Brute-forcing `userId` to discover legitimate IDs is **not** a real risk ‚Äî IDs are 9-10 digit integers, sparsely allocated, and the reply only echoes the *sender's own* id back to them, so no information is leaked about other accounts. The risk here is DoS, not auth bypass.

**Mitigations, choose any:**

- In-memory rate limit per chat_id (e.g. 1 `/start` per 60s per chat).
- Stop responding to `/start` from a chat once it has produced a successful reply once (remember in cursor file).
- Optional `--strict` mode that ignores all non-whitelisted senders entirely (no reply).

### üüÝ Substring matching in `requirePlan`

```ts
const full = plans[planId] ?? Object.values(plans).find((p) => p.id.startsWith(planId));
```

If the user types `/kill ab` and two plans start with `ab‚Ä¶`, only the first found is killed, non-deterministically (depends on `Object.values` order, which is insertion order ‚Äî usually stable, but easy to forget). For `/show` and `/notes` the worst case is showing the wrong plan; for `/kill` it can kill the wrong fleet. Either require a minimum prefix length (e.g. 6 chars) and refuse shorter, or list ambiguous matches and abort.

### üüÝ `/kill` doesn't validate the target task is killable

`/kill <plan> <task> <reason>` will happily set `stopRequested` on a task whose status is already `done`, `failed`, `cancelled`, or `pending`-with-no-session. The CLI's `kill_task` MCP tool likely has this check; the Telegram path bypasses it. Mirror the validation here ‚Äî at minimum, reject if `task.status` is terminal.

### üü° `cfg` is loaded once per command ‚Äî owner change is visible immediately

Not a bug, just calling it out: every command does `await loadConfig()` so any whitelist change applied externally takes effect on the next command. Good.

### üü° `cmdTgShow` token preview

`src/cli.ts:233`: `cfg.botToken.slice(0, 8) + "‚Ä¶"`. Telegram tokens are `<bot_id>:<35-char-secret>` and the first 8 chars are always the (public) bot-id digits, so this is safe ‚Äî but I'd reduce to 6 to be extra cautious, or print only the bot id (everything before the `:`).

## `src/telegram/notifier.ts`

### üî¥ First-tick replay flood

When the bot starts on an existing project, `loadCursor` returns `EMPTY_CURSOR` (no `taskStatus`/`planStatus` map). On the first tick:

- Every plan: `prevStatus = undefined`, so `prevStatus !== plan.status` is true ‚Üí if status is `approved`, `collected`, `cancelled`, or `cancelled_overbudget`, a message fires.
- Every task whose status is currently `done`/`failed`/`cancelled` produces a message.

For an active project with 5 completed plans √ó 4 tasks each, that's 20+ messages on first start. The notes path has a backfill (lines 124-127: seed `lastNoteId` to the latest); statuses do not.

**Fix:** mirror the notes pattern. On the very first tick (detected by `cursor.taskStatus` and `cursor.planStatus` both being empty), seed both maps with current statuses *without* emitting messages. Subsequent ticks behave normally.

### üüÝ Cursor-advance-before-send loses messages permanently

Documented as a deliberate tradeoff in lines 22-23 and the commit message ("Cursor advances BEFORE send so a failed network call doesn't re-spam old events. Tradeoff: occasionally a notification can be missed; acceptable for this surface."). I agree with the directional choice ‚Äî duplicate-spam is worse than miss ‚Äî but combined with the Markdown injection bug above this becomes a **silent and permanent** drop for any plan with `_` in its `rootTask`.

A small improvement: keep a bounded retry queue (in the same cursor file) of e.g. the last 16 failed messages. On the next tick, try them again before advancing. This recovers from transient network errors without re-blasting on app restart. It's optional.

### üü° Per-tick allocation of `StateStore`

`tick()` instantiates `new StateStore(...)` every 5 s, defeating the in-memory cache. Performance-wise this is fine, but if the cache is supposed to be the only consistency anchor for in-process callers, it's a footgun. (See concurrency note above ‚Äî the cache isn't actually a consistency anchor across processes, so the new instance is the only correct choice here.)

### üü° No bound on `messages.length` per tick

If a plan suddenly emits 50 new notes (e.g. an agent dumps a transcript via `post_note`), the notifier will issue 50 sequential `sendMessage` calls per whitelisted user, with no batching and no rate-limit awareness. Combine that with Telegram's ~30 messages/sec per-bot global limit and you can self-429. Either coalesce per-plan note bursts into a single message ("3 new notes from `task-xyz`‚Ä¶") or sleep ~100 ms between sends.

## `src/telegram/bot.ts`

### üüÝ `SIGINT` handler kills mid-write

```ts
process.on("SIGINT", () => {
  stopFlag.stopped = true;
  process.stdout.write("\norqlaude tg: shutting down.\n");
  process.exit(0);
});
```

`process.exit(0)` is synchronous and immediately tears down the event loop. Consequences:

- `stopFlag.stopped = true` is irrelevant ‚Äî neither loop ever reads it again.
- The `await notifierTask` at line 72 is unreachable code.
- If a `/kill` is in the middle of `stateStore.update()` (between `writeFile(tmp)` and `rename(tmp, ‚Ä¶)`), the tmp file is left orphaned.

**Fix:** drop `process.exit` from the handler; just set the flag and let the loops drain. Add `AbortController` to `getUpdates` so the long-poll wakes up immediately. The `await notifierTask` then actually means something. Track orphan `.tmp` files with `fs.glob` on startup and delete them.

### üü° Single SIGINT handler ‚Äî double Ctrl-C does nothing extra

Not a bug yet, but as soon as you remove `process.exit`, users will want a double-Ctrl-C "force-quit" path. Cheap to add: on second SIGINT, `process.exit(1)`.

### üü° `await notifierTask` after `process.exit` is dead code

See above ‚Äî also a TypeScript/eslint smell that's worth fixing now to avoid confusion.

## CLI wiring (`src/cli.ts` `cmdTg*`)

### üü° `cmdTgSetup` doesn't trim or validate token shape

A bot token looks like `\d+:[A-Za-z0-9_-]{35}`. Currently any non-empty string is accepted and the only validation is the `getMe` round-trip. That's *good enough* (the network call is the real authority), but adding a regex sanity check before the HTTP call would catch the common paste-with-trailing-newline / quoted-token mistakes earlier with a friendlier message.

### üü° `cmdTgTest` exits 1 with stderr on Markdown failure too

Minor: `api.sendMessage` from `cmdTgTest` uses no `parseMode` so it's safe, but if you later add a `--markdown` flag, the test path should print a clearer hint when the failure is "your text broke the parser" vs "bot is offline."

### üü° `cmdTgStart` runs forever in foreground only

The bot has no daemonization path ‚Äî `tg start` blocks the terminal. For a "personal fleet" tool this is probably correct (encourages tmux/launchd ownership), but the README's walkthrough should explicitly mention `launchd`/`systemd`/`tmux` as the supported deployment shape. Right now it reads like a one-shot.

## Things explicitly checked that look fine

- **`isAuthorized`** correctly accepts owner OR whitelisted user; both paths gated.
- **`/whitelist` owner check** correctly rejects non-owner senders before any state change.
- **Notifier note backfill** (lines 124-127) prevents spam on first run for notes specifically.
- **`/kill` reason defaulting** (`"killed from Telegram"`) is fine.
- **Token never appears in CLI args, env, or process title.**
- **`getUpdates` `offset = u.update_id + 1`** correctly advances past consumed updates so we don't re-process on bot restart.
- **`/start` reply does not leak any other user's info** ‚Äî only echoes the sender's own ID. No brute-force enumeration angle.
- **In-process `notifier` ‚Üî `handleCommand` race** is benign ‚Äî `notifier.tick` only reads, `handleCommand` serializes per-update via `await`. No deadlock risk because the in-memory lock is per-instance and not re-entrant or shared.

## Recommendation

I'd block merge to "documented stable" on the four üî¥ items:

1. `chmod 0o600` race in `saveConfig` (10-line fix).
2. Markdown escaping (1-2 hours; high impact on perceived reliability).
3. Cross-process `StateStore` mutation safety for `/kill` ‚Äî a 30-line `proper-lockfile`-style fix is enough; SQLite can wait.
4. First-tick status replay flood.

The üüÝ items are quality-of-life and should land in v0.3.1.

The architecture itself ‚Äî zero-dep, long-poll, cursor file, mode-600 config, message-on-delta ‚Äî is the right shape for this surface. None of the issues above require structural changes.
