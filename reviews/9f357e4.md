## [BLOCKER] `.mcp.json.template` does not start the MCP server

`.mcp.json.template` instructs end-users to install via:

```json
{
  "mcpServers": {
    "orqlaude": {
      "command": "npx",
      "args": ["-y", "@synaplink/orqlaude"]
    }
  }
}
```

But `package.json` defines:

```json
"main": "./dist/server.js",
"bin":  { "orqlaude": "./dist/cli.js" }
```

`npx -y @synaplink/orqlaude` resolves to the package's bin (the unscoped name `orqlaude` matches the bin key), so it executes `dist/cli.js` — the read-only inspection CLI — not `dist/server.js`. With no subcommand, `cli.ts` prints the help banner to stdout and exits `0`. Claude Code's stdio MCP transport then sees a clean exit with no JSON-RPC handshake and the server "starts" but registers zero tools.

**Net effect:** anyone who follows the README + template ends up with a registered MCP that doesn't speak MCP. `npm` is irrelevant — `main` is not used by `npx`.

**Suggested fix (one of):**

1. Add a second bin entry pointing at the server, and update the template:
   ```json
   "bin": {
     "orqlaude":      "./dist/cli.js",
     "orqlaude-mcp":  "./dist/server.js"
   }
   ```
   ```json
   "args": ["-y", "@synaplink/orqlaude", "orqlaude-mcp"]
   ```
2. Or have `cli.ts` dispatch to the server when invoked with no args / with a `serve` subcommand, and call that explicitly from the template.

Option (1) is cleaner — the CLI's stdout/help shouldn't share a process role with a JSON-RPC stdio server.

## [BLOCKER] `dist/cli.js` is not executable in the published tarball

`tsc` does not preserve the executable bit on emitted files (it has no concept of file permissions on the source). The shebang is in `src/cli.ts`, but the resulting `dist/cli.js` is mode `0644` after a fresh `tsc`. npm requires bin files to be executable; on POSIX, `npm i -g` will fail with `permission denied`.

The task brief notes this was patched after publish. The fix should land in this repo, not just in the registry tarball. Add a `postbuild` hook so the source-of-truth artifact is correct:

```diff
   "scripts": {
     "build": "tsc",
+    "postbuild": "chmod +x dist/cli.js",
     "dev": "tsc --watch",
```

(Cross-platform note: `chmod` is a no-op on Windows, which is fine — npm there ignores the bit and uses `cmd-shim`.)

## [CONCERN] `review_prs` does not validate `pr_url`; assumes GitHub

`src/tools/review.ts:90-99` builds the reviewer prompt by interpolating `pr_url` directly into shell commands and Markdown:

```ts
1. Fetch the PR locally: `gh pr checkout <number>` or `gh pr view ${prUrl} --json files,title,body`.
...
6. If you find a [BLOCKER], also leave a GitHub review comment via `gh pr review <pr> -c -b "..."`
```

Two issues:

1. **No URL validation.** `prUrl` is whatever an arbitrary fleet child wrote into `post_note({ pr_url: ... })`. If a malformed value (or one containing backticks / `$(...)`) makes it into a shell command the reviewer agent then runs, the child has effectively achieved arg-injection on the orchestrator host. The risk is bounded (Claude Code agents don't usually mechanically obey injected shell, and `pr_url` came from another fleet member, not random internet input), but defense-in-depth is cheap here:

   ```ts
   function isLikelyGithubPrUrl(u: string): boolean {
     try {
       const url = new URL(u);
       return url.host === "github.com"
         && /^\/[^/]+\/[^/]+\/pull\/\d+\/?$/.test(url.pathname);
     } catch { return false; }
   }
   ```
   Reject in the `update` mutator and surface a `note: "skipped: bad pr_url for task X"` rather than dispatching a doomed reviewer.

2. **GitHub-only.** The prompt hard-codes `gh pr ...`. Repos hosted on GitLab/Bitbucket/Gitea/Codeberg have no `gh`-compatible flow. Either document the constraint, or detect-and-skip non-`github.com` URLs with a clear note in the result payload.

## [CONCERN] CLI: `cmdStatus` does no I/O batching and doesn't truncate task titles

`src/cli.ts:99-111`:

```ts
for (const t of plan.tasks) {
  let extra = "";
  if (t.spawnedSessionId) {
    const snap = await snapshotSession(process.cwd(), t.spawnedSessionId);
    ...
  }
  console.log(`    [${t.status.padEnd(10)}] ${t.title.padEnd(50)}${extra}`);
```

- `snapshotSession` is awaited sequentially per task. With 10 tasks each tailing a JSONL file, latency is additive. `Promise.all(plan.tasks.map(...))` would parallelize trivially.
- `t.title.padEnd(50)` does **not** truncate when `title.length > 50`, so any longer-than-50-char title breaks the column alignment for the tokens column. `cmdList` does truncate (`truncate(p.rootTask, 60)`) — be consistent: `truncate(t.title, 50).padEnd(50)`.

The `truncate(s, n)` helper itself is safe for very long strings (slice + "…"), but is degenerate for `n <= 1` (`slice(0,0) + "…"` = `"…"`). No callers pass small `n`, so it's fine — just worth a guard if it ever gets reused.

**Graceful-on-missing-state check passes:** `StateStore.load` and `AuditLog.tail` both swallow `ENOENT` and return empty defaults (`src/lib/state.ts:128`, `src/lib/audit.ts:108`), so `orqlaude list` and `orqlaude history` print "No plans yet" / "(no audit events yet)" on a fresh project. `status`/`show` propagate `Plan not found` and exit `1`. Good.

## [CONCERN] CI is single-platform, single-Node, no audit gate

`.github/workflows/ci.yml`:

- **Pinned to `node-version: "24"`** — major-only pin. Reasonable, but consider a matrix `["22", "24"]` since `engines.node: ">=22"` is what the package advertises. Otherwise you can ship a v0.2.x that quietly breaks on 22.
- **`ubuntu-latest` only.** The `normalizeClaimPath` test asserts `/repo/root/src/foo.ts` — POSIX-style. On Windows, `path.normalize` returns `\repo\root\src\foo.ts` and the test would fail. Either constrain the package to POSIX in `engines.os` or add a `runs-on: [ubuntu-latest, macos-latest]` matrix (Windows is a bigger lift; macOS at minimum is cheap and guards against case-insensitivity surprises in claim paths).
- **`cache: "npm"` is set** — good.
- No `npm audit` / `npm outdated` step. Optional, but for a published package it's worth a low-severity gate.

## [CONCERN] Test coverage gaps

The 11 tests focus on `state.ts` and `hallucination.ts`. State-mutation paths that aren't exercised — and which the audit log tells me are the most-frequently called tools:

- **Budget enforcement.** No test that crossing `budgetCapTokens` flips the plan to `cancelled_overbudget` and emits stop signals.
- **`kill_task` / stop signal pickup.** `stopRequested` field exists on `Task` but there's no test that a checkin after `kill_task` returns the stop directive.
- **File claims.** `normalizeClaimPath` is tested, but the claim/release flow (conflict on already-claimed path, release by non-owner) isn't.
- **Broker.** `send_message`/`poll_notes` round-trip and the `kind: "stop"` semantics.
- **`review_prs` itself has zero tests.** Easy to add one: build a plan with 2 tasks (one with `prUrl`, one without), call the tool, assert review plan has 1 task and source plan got `reviewPlanId` set, and that `auto_approve: false` leaves it `draft`.
- **CLI** has zero tests. A small smoke test (`exec("node dist/cli.js list", { env: { ORQLAUDE_STATE_DIR: tmp } })` returning 0) would catch shebang/permission/import-path regressions.

The hallucination tests are reasonable for the current detector behavior; they don't test the read-then-edit-different-file case which is the most likely false-positive surface.
