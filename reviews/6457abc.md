## [BLOCKER] `StateStore.read()` bypasses the write lock — torn-state reads are reachable

`src/lib/state.ts:122-126`:

```ts
async read<T>(reader: (state: State) => T): Promise<T> {
  const state = await this.load();
  return reader(state);
}
```

`update()` serializes via `this.writeLock`, but `read()` does not await it. The mutator pattern is:

```ts
const state = await this.load();           // state === this.cache
const result = await mutator(state);        // mutates the SAME object in-place
await this.persist(state);                  // async, fs.writeFile + fs.rename
```

If a `read()` lands between `mutator()` returning and `persist()` resolving, the reader sees an in-memory mutated state that is not yet on disk. If the mutator is itself async (the signature allows `T | Promise<T>`), readers can observe partial mutation while awaits are pending. None of the mutators registered in this commit are async, so the practical window is small — but it exists and is a footgun for the next maintainer who writes `async (state) => { ... await ... }`.

This is compounded by a second issue in the same `update()` (`src/lib/state.ts:108-120`): if `mutator()` throws, `this.cache` has already been mutated in place, no rollback runs, and the next reader sees the broken state. There is no try/catch that restores the prior cache.

Suggested fix:
- Funnel `read()` through the same `writeLock` chain (read can run on the same queue, or use an rwlock).
- Wrap the mutator in a try/catch that restores `this.cache` from the pre-load snapshot on throw (deep-clone before mutating, or re-load from disk on failure).
- At minimum, document loudly that mutators must be synchronous and side-effect free.

## [BLOCKER] Audit `summarize()` leaks the approval_token even though args are redacted

`src/lib/audit.ts:108-118` (the result summarizer used by every wrapped tool):

```ts
function summarize(result: unknown): string {
  try {
    const r = result as any;
    const text = r?.content?.[0]?.text;
    if (typeof text === "string") return text.slice(0, 200);
    return JSON.stringify(result).slice(0, 200);
  } catch { ... }
}
```

`redactSecrets()` redacts `approval_token` from the **args** at `src/lib/audit.ts:120-129`, but the `request_approval` tool returns the freshly-minted token in `result.content[0].text` (`src/tools/planning.ts:144-148`):

```ts
text: JSON.stringify({
  approval_token: plan.approvalToken,
  summary: ...,
  ...
}, null, 2)
```

JSON.stringify with `null, 2` makes `approval_token` the first or near-first field, well within the first 200 chars that `summarize()` keeps. So `.orqlaude/audit.jsonl` contains the cleartext token in `resultSummary` for every `request_approval` call. Approval tokens are single-use and cleared on `confirm`, which mitigates real-world risk — but the redactor's stated job (don't sit tokens in the log forever) is defeated by the result-side leak.

Fix: redact known sensitive keys in `summarize()` too — or run the result text through a regex scrub for `approval_token`/UUID patterns — or just don't put the token in the user-facing `content.text` at all (return it only via a side channel).

While in there: redaction is shallow (`"approval_token" in clone`) and only top-level. New secret-bearing fields (e.g., a future webhook URL with auth) would slip through silently. An allowlist or a recursive walker would be safer.

## [CONCERN] v1→v2 migration synthesizes `claims` but not `notes` / `messages`

`src/lib/state.ts:160-174`:

```ts
out.plans[id] = {
  ...p,
  budgetCapTokens: p.budgetCapTokens ?? Math.round((p.budgetCapUsd ?? 5) * 25_000),
  perAgentCapTokens: p.perAgentCapTokens ?? Math.round((p.perAgentCapUsd ?? 1) * 25_000),
  claims: p.claims ?? [],
} as Plan;
```

`claims` gets a default (good — it's new in v2). `notes`, `messages`, and `tasks` do not. If any v1 plan in the wild was persisted without those arrays (early development, partial corruption, hand-edited state), the first call into `plan.messages.push(...)` (`src/tools/broker.ts:286-294`, `src/tools/dispatch.ts:128-136`, `src/tools/lifecycle.ts:48-56`) throws `Cannot read properties of undefined (reading 'push')` and the whole tool call fails. The migration's job is to make v1 state safe for v2 code — the defensive defaults belong here, even if "v1 always had them" in theory.

Also: `Math.round((p.budgetCapUsd ?? 5) * 25_000)` silently caps a budgetless v1 plan at 125k tokens. Once that v1 plan is reloaded under v2, the next `status()` likely flips it to `cancelled_overbudget` and queues STOP messages with no explanation a user could trace. At minimum, log a warning during migration; better, refuse to migrate a plan that has neither USD nor token caps and surface it via `list_plans`.

Malformed-input check: `Object.entries(input.plans ?? {})` handles `undefined` and `null` but throws TypeError if `plans` is a string/number/array. No structural validation of each plan body (a corrupt `tasks: null` would break downstream code). Worth a `Zod` parse at load time, or at least a `Array.isArray(p.tasks)` guard.

## [CONCERN] JSONL tail cache invalidates on size shrink only — rotation / replay returns stale snapshots

`src/lib/jsonl_tail.ts:85-92`:

```ts
let entry = cache.get(jsonlPath);
if (entry && entry.byteOffset > stat.size) { cache.delete(...); entry = undefined; }
if (entry && entry.byteOffset === stat.size) return entry.snap; // no new data
```

The invariant is "size strictly grows → cache valid". Counterexamples:

1. **Rotation to identical size.** If the file is deleted and replaced (or truncated and re-written) to exactly the previous size, neither condition fires. The cache returns the prior snapshot indefinitely. With Claude Code session resets in the same project, this is plausible.
2. **Inode change.** `fs.stat` doesn't compare inode; a fresh file at the same path with a different inode is treated as continuation.
3. **Same-session restart.** A re-opened session JSONL likely won't hit exact byte parity, but a deterministic replay or test fixture could.

Add `stat.ino` and `stat.mtimeMs` to the cache key/check. The cost is one extra stat field; the upside is correctness under file rotation.

While here: the cache returns `entry.snap` directly (`src/lib/jsonl_tail.ts:91`). Callers that mutate the returned snap mutate the cache. Not currently exploited but a real footgun (e.g., a future caller adding `if (snap.terminated) snap.terminationReason = '...'`).

UTF-8 boundary: `buf.toString("utf8")` over `Buffer.alloc(stat.size - startOffset)` (`src/lib/jsonl_tail.ts:97-105`) decodes the bytes once, then carries the trailing partial line as a STRING. If a multi-byte char (CJK, emoji) is split across a read boundary, decoding inserts U+FFFD into the carry; the next read concatenates more decoded text and the JSON line will fail to parse silently. Rare in practice but a real lossy-decode bug. Fix: carry bytes (`Buffer`) instead of a `string`, decode after concatenation.

## [CONCERN] `extractToolUses` re-reads the full JSONL on every `status()` — defeats the byte-offset cache the commit advertises

`src/lib/hallucination.ts:154-175`:

```ts
export async function extractToolUses(jsonlPath: string): Promise<ToolUseEvent[]> {
  const raw = await fs.readFile(jsonlPath, "utf8");      // full file every call
  ...
}
```

…and `src/tools/dispatch.ts:90-95`:

```ts
const snap = await snapshotSession(cwd, t.spawnedSessionId);
const toolUses = await extractToolUses(jsonlPathFor(cwd, t.spawnedSessionId));
```

The commit message advertises the JSONL byte-offset cache as the headline status() optimization (O(filesize) → O(new-bytes)). Two lines later, `extractToolUses` re-reads the whole file for every agent on every status() call to feed the hallucination detector. With a fleet of 5 agents and multi-MB JSONLs, that's the entire optimization eaten back. The `snapshotSession` cache already tracks `lastToolUse` (only the latest); a parallel cache for the full tool_use history — populated incrementally by the same `applyEvent` walk — would keep the win.

Cheap fix: extend `CacheEntry` (or a sibling cache) to accumulate `ToolUseEvent[]` as `applyEvent` sees them; `extractToolUses` then reads from that cache, doing a delta read only if the byte offset advanced. The downside is unbounded memory growth — apply a sliding window (last N events).

## [CONCERN] Hallucination detector — specific false positives and false negatives

`src/lib/hallucination.ts:106-119` (tight-loop):

```ts
if (maxRepeat >= 4) {
  score += 0.2;
  concerns.push(`Repeated the same tool call ${maxRepeat} times — agent may be looping.`);
}
```

A legitimate polling loop trivially clears 4 — e.g., `Bash: gh pr checks <id>` every 30s while a CI run is in flight, or `Bash: ls .orqlaude/audit.jsonl` while watching. The detector cannot tell that apart from a stuck loop. Consider:
- Exempting read-only Bash commands matching a polling allowlist (`gh pr checks`, `git status`, `ls`, `cat`-on-state-file), or
- Requiring the repeated call to also have produced an error/no-progress signal, or
- Looking at *consecutive* repeats vs total-call repeats (a 4× repeat spread across 50 turns is fine; 4× consecutive is suspicious).

`src/lib/hallucination.ts:121-133` (commit-without-test):

```ts
const tested = bashCmds.some((c) =>
  /\b(npm\s+test|pytest|cargo\s+test|tsc|lint|check|jest|vitest|mocha)\b/.test(c)
);
```

Missing: `yarn test`, `pnpm test`, `bun test`, `go test`, `make test`, `gradle test`, `dotnet test`, `python -m pytest`, `./run_tests.sh`. Several of these are common in this fleet's likely targets. Result: false-positive "Committed without running tests" on Go/Bun/etc. projects. Either broaden the regex (`(npm|yarn|pnpm|bun)\s+(test|check)`, `(go|cargo|make|gradle|dotnet)\s+test`, `python\s+-m\s+pytest`) or invert it (look for known-NOT-test patterns).

`src/lib/hallucination.ts:84-101` (edit-without-prior-read):

```ts
if (READ_TOOLS.has(tu.name)) {
  const p = extractPathFromToolUse(tu);
  if (p) readPaths.add(p);
}
```

`extractPathFromToolUse` only handles `file_path` / `path` / `notebook_path`. The Grep and Glob tools use `pattern` (with `path` as a directory root, not the file). So `Grep("export const foo", path: "src/")` followed by `Edit("src/lib/foo.ts", ...)` flags as "edited without prior read". Also, a `Bash: cat src/foo.ts` followed by an Edit flags too — and that pattern is endemic. Conservative fix: don't add `+0.15` for paths that match a recent Grep root directory or a Bash `cat`/`head`/`tail`/`less` argument.

`src/lib/hallucination.ts:60-66` (path-existence cwd check):

```ts
if (!p.startsWith(cwd) && !p.startsWith("/tmp")) continue;
```

`cwd` here is the orqlaude SERVER's cwd (`process.cwd()` in `status()`), not the spawned agent's worktree. Today this works because worktrees live at `<server-cwd>/.claude/worktrees/<name>/...` and so `startsWith(cwd)` is true. But anyone running orqlaude with `ORQLAUDE_STATE_DIR` outside the project, or with a child whose worktree is somewhere else, gets silent skipping (every path filtered out → 0 missing → 0 score). Better: pull each agent's actual cwd from the JSONL's session record and check against that.

TOCTOU on `fs.access`: the detector runs AFTER the tool call ran. If the agent created and then removed a file (rm/git clean/test artifact), the path is flagged as missing even though the agent's behavior was correct. Hard to fully solve, but checking event order — was the path created within the same session before being removed? — would suppress this class.

## [CONCERN] Auto-cancel-on-overbudget is idempotent in practice, but the guard is racy

`src/tools/dispatch.ts:118-141`:

```ts
const overbudget = totalTokens > plan0.budgetCapTokens;
let autoCancelled = false;
if (overbudget && plan0.status !== "cancelled_overbudget" && plan0.status !== "cancelled") {
  await store.update((state) => {
    const plan = findPlan(state, plan_id);
    plan.status = "cancelled_overbudget";
    for (const t of plan.tasks) {
      if (t.spawnedSessionId && !t.stopRequested) {
        t.stopRequested = { reason: "fleet overbudget", requestedAt: Date.now() };
        plan.messages.push({ ... kind: "stop", ... });
      }
    }
  });
  autoCancelled = true;
}
```

`plan0` is read OUTSIDE the write lock. Two concurrent `status()` calls can both pass the guard, both call `store.update`. The mutator does NOT re-check `plan.status` or `overbudget` — it unconditionally re-sets status to `cancelled_overbudget` and re-iterates tasks. Today, double-queueing of STOP messages is prevented because the per-task `!t.stopRequested` check skips already-flagged tasks. So idempotency holds — by accident, not by design.

If a future maintainer changes the per-task guard, double-queue returns. Hardening:
1. Move the `overbudget && !already-cancelled` check INSIDE the mutator (so it's checked against fresh state under lock).
2. If not over budget anymore or already cancelled, return early from the mutator and skip the redundant `persist()` write.

Also: this depends on `read()` returning a fresh snapshot (see BLOCKER above) — `plan0.status` aliases the live cache, so by the time we compare it, another writer may have already flipped it. The aliasing currently doesn't bite because the mutator handles it, but it's fragile.

## [CONCERN] `cryptoRandomId()` is not crypto-strong, name implies otherwise

`src/tools/dispatch.ts:236-239`:

```ts
function cryptoRandomId(): string {
  // small helper to avoid importing crypto in this top-of-file scope twice
  return Math.random().toString(36).slice(2) + Date.now().toString(36);
}
```

The comment explains the laziness, but the name is actively misleading — anyone reading `cryptoRandomId()` in a security-adjacent codebase will assume CSPRNG. The function is used for in-process message IDs (`plan.messages.push({ id: cryptoRandomId(), ... })`), which is fine functionally, but if anyone reuses it for an approval token or claim token it's a real vuln. Either rename to `messageId()` or just use the already-imported `randomUUID` from `node:crypto` (everywhere else in the codebase does — `src/tools/broker.ts:6`, `src/tools/lifecycle.ts:3`, `src/lib/state.ts:3`). The duplication concern in the comment is unfounded; one import is fine.

## [NIT] Misc

- `src/lib/state.ts:122-126` — `read()` returns the live state reference. Callers like `dispatch.ts:78` (`const plan0 = await store.read(...)`) hold a Plan reference that subsequent `update()` calls mutate in place. Today the aliasing happens to mask the bug correctly (status() reads `autoCancelled ? "cancelled_overbudget" : plan0.status` and gets the right answer). It's a landmine for the next reader. Either return a `structuredClone` or document that `read()` returns a live reference.
- `src/lib/state.ts:103` — `private cache: State | null = null;` and `this.cache = state` on every persist. The cache is mutated in place by mutators, but `persist()` is the only point it's reassigned — meaning re-assignment is a no-op when `mutator` already mutated the same reference. Harmless but confusing.
- `src/lib/audit.ts:38-49` — `this.writeQueue = this.writeQueue.then(...)`. If the chained promise rejects (the catch block re-stringifies and writes stderr — won't reject), nothing breaks. But there's no bound on the queue depth; under heavy load, the unresolved chain holds references to every queued event. In practice not an issue.
- `src/lib/budgeting.ts:73` — `budgetCapUsd = budgetCapTokens / 25_000` is a fixed conversion (~Sonnet blended rate). For Haiku-driven fleets the USD shadow is off by an order of magnitude. Fine since USD is "informational", but the misleading number is visible in `request_approval`'s prompt to the user.
- `src/lib/state.ts:254-258` (`normalizeClaimPath`) — `path.normalize` does not case-fold. macOS default filesystems are case-insensitive; `/Foo/Bar.ts` and `/foo/bar.ts` would yield different claims for the same on-disk file. Two agents could "claim" the same file with different casings and neither sees a conflict.
- `src/tools/dispatch.ts:80-110` — every `status()` does `Promise.all(plan0.tasks.map(...))`, each entry runs `snapshotSession` + `extractToolUses` + `detectHallucination`. For a 12-task plan with multi-MB JSONLs, this is a lot of disk I/O per poll. Worth a brief delay / debounce in the caller, or memoizing per-call.
- `src/tools/planning.ts:30` — `DEFAULT_BUDGET_TOKENS = 500_000` is reasonable, but `confirm` doesn't enforce that `estimatedTokens <= budgetCapTokens`. A user can approve a plan whose estimate already exceeds the cap (only `estimate()` returns `fits_budget` informationally). Either reject in `confirm` or surface a louder warning in `request_approval`.
- `src/tools/broker.ts:268-281` — `send_message` does not validate that `to_session_id` corresponds to a known task in this plan. Typo'd IDs queue undeliverable messages forever (cleaned only by plan lifetime).
- `src/tools/lifecycle.ts:128-148` (`list_plans`) — no cap on `plans` array length. After hundreds of plans across the lifetime of a project, the response grows unbounded. Add `limit` (default 20) and order by recency.
- `src/server.ts:35` — `await server.connect(transport)` has no error handler. A transport failure surfaces as unhandled rejection / process exit with no diagnostic.

## What I verified explicitly (not just "looks fine")

- **Migration on v2 input**: `if (v === 2 && input.plans) return input as State` — short-circuits correctly when both conditions hold. If `input.plans` is `undefined`, falls through to the v1 path and produces an empty-plans State — acceptable. If it's `null`, the falsy check still falls through, and `Object.entries(null)` would actually throw — minor, edge case.
- **Approval-token consumption** in `confirm` (`src/tools/planning.ts:166-184`): token cleared via `plan.approvalToken = undefined` after match. Re-confirm with the same token throws "Approval token mismatch". Single-use semantics hold.
- **`unclaimedTaskById`** (`src/lib/state.ts:239-244`): correctly filters on `!t.spawnedSessionId` so already-self-registered tasks aren't double-claimed by a second checkin with the same task_id.
- **STOP message wiring**: dispatch.ts overbudget path → broker.checkin filtering messages where `m.kind === "stop"` → returns as `stop_signal` and removes from `messages` array delivered to the agent (`src/tools/broker.ts:75-87`). Wiring works end-to-end. Idempotency holds today (see CONCERN section).
- **Audit write serialization**: `this.writeQueue = this.writeQueue.then(...)` chain ensures appendFile calls don't interleave even with N concurrent tool calls. Verified against the 11 passing tests + by reading the chain construction.
- **`buildSpawnPrompt`** correctly embeds both `plan_id` and `task_id` and instructs the child to call `checkin` first. Matches `checkin`'s self-registration logic in broker.ts.

## Top recommendations (in order)

1. Fund the BLOCKER fixes first: read-side write-lock + audit-result redaction.
2. Default `notes`/`messages`/`tasks` in the v1→v2 migration; warn loudly on synthesized budgets.
3. Add `stat.ino` + `stat.mtimeMs` to the JSONL cache key/check.
4. Cache `extractToolUses` incrementally so the headline status() optimization actually applies.
5. Tighten the hallucination heuristics (yarn/pnpm/bun/go test patterns; consecutive-repeat check; grep/cat as evidence of "read").
6. Rename or replace `cryptoRandomId()`.
